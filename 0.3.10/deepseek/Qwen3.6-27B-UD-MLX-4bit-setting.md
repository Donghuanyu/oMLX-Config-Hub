# Qwen3.6-27B-UD-MLX-4bit 最佳模型设置

> **生成平台**：OMLX 0.3.10  
> **运行硬件**：Mac mini 4 M4（内存 64GB 统一内存，SSD 2TB）  
> **使用场景**：Java 后端开发 / 前端开发 / 小程序开发 / 代码重构

---

## 1. 模型基本信息

| 属性 | 值 |
| :--- | :--- |
| **模型名称** | Qwen3.6-27B-UD-MLX-4bit |
| **模型架构类型** | **稠密型模型 (Dense)** |
| **参数量** | 27B（全部参数每次前向传播均被激活） |
| **4bit 量化后内存占用** | 约 17~18 GB |
| **架构说明** | 传统的全参数稠密模型。每次 Token 生成都需要加载全部 27B 权重进行矩阵乘法运算，对统一内存带宽的压力恒定且巨大。但由于参数全部被激活，其逻辑推理连贯性极强，代码生成质量稳定，特别适合作为主力开发模型。 |

---

## 2. 推理加速技术深度解析

### 2.1 DFlash 块扩散投机解码 — 推荐：✅ **开启**

| 项目 | 说明 |
| :--- | :--- |
| **为什么需要** | 稠密模型的内存带宽瓶颈最为严重。每次解码都需从统一内存读取全部 27B 权重。DFlash 通过引入小型草稿模型一次性预测多个候选 Token，再由主模型并发验证，可将内存读取次数减少 3~4 倍，极大突破带宽限制。 |
| **核心优点** | ① **3~4x 端到端解码速度提升**，显著降低开发过程中的等待感；② **无损精度**，主模型验证保证输出质量与直接生成完全一致；③ **支持回退机制**，超长上下文自动回退标准解码，不会崩溃。 |
| **核心缺点** | ① 需额外加载草稿模型（约多占 3~5 GB 内存），但在 64GB 统一内存上完全可以承受；② 草稿模型上下文窗口有限（4096 Token），超出后自动降级；③ 仅支持单请求流，适合单人开发场景。 |
| **Draft 模型推荐** | `z-lab/Qwen3.6-27B-DFlash` — 官方提供的真实 DFlash 草稿模型，可从 HuggingFace 下载。 |

### 2.2 SpecPrefill 投机预填充 — 推荐：❌ **关闭**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 用极小的模型快速计算超长 Prompt 的预填充阶段，降低首字延迟（TTFT）。 |
| **为何关闭** | ① 与 DFlash 存在功能重叠且相互冲突；② 短 Prompt 下无加速效果反而增加额外开销；③ 需额外加载一个小模型，挤占 DFlash 草稿模型的可用内存。在已开启 DFlash 的前提下，SpecPrefill 收益甚微。 |

### 2.3 Native MTP 原生多 Token 预测 — 推荐：❌ **关闭（当前不可用）**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 利用模型内置的 MTP 预测头一次性预测多个未来 Token，无需额外加载草稿模型，可实现约 1.5x 稳定提速。 |
| **当前状态** | **不可用**。您下载的 4bit 量化版本在 MLX 转换过程中遗失了 `mtp.*` 权重张量（`mtp.layers.*`），OMLX 加载时会提示 `Config declares MTP layers but the converted weights are missing mtp.* tensors` 错误。 |
| **启用指南** | ① 下载带 MTP 头的 Qwen 官方 HF 原生仓库（未来 Qwen 3.6 若有带 `-MTP` 后缀的版本，请选此类）；② 使用最新版合并了 PR #990 和 PR #15 补丁的 `mlx-lm` 重新进行 4bit 量化转换：`python -m mlx_lm.convert --hf-path <官方含MTP权重的HF仓库> -q --q-bits 4 --mlx-path <输出路径>`。 |

## 3. OMLX 0.3.10 模型设置参数列表

### 4.1 模型类别
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **模型类别** | `llm` | 强制指定为纯文本大语言模型，避免自动检测误判为 VLM 或 Embedding 等其他类别，确保工具调用格式稳定。 |

### 4.2 REASONING PARSER
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **REASONING PARSER** | `qwen_3_coder` | Qwen 3.6 属于 Qwen3-Coder 系列，选用 `qwen_3_coder` 解析器可正确截取模型的推理链输出结构，防止非推理内容被误解析。 |

### 4.3 ~ 4.11 采样与输出参数
| 配置项 | 推荐设定值 | 针对防假死、防死循环与输出质量的设计理由说明 |
| :--- | :--- | :--- |
| **4.3 Ctx Window** | `32768` | 64GB 统一内存下完全可行。32K Token 上下文可容纳完整的大型项目调用链和多个文件上下文，避免频繁的上下文截断导致模型丢失关键信息。 |
| **4.4 最大 Token 数** | `4096` | 限制单次最大输出长度。防止模型因发散导致无限输出、假死或返回超长无意义文本，同时确保工具调用的响应及时返回。 |
| **4.5 温度** | `0.2` | 低温度确保高确定性输出，对代码生成尤其重要。温度过高会导致模型"创意发散"产生不符合语法的代码或胡言乱语。 |
| **4.6 Top P** | `0.9` | 剔除累积概率 10% 以外的低概率 Token，在低温度下辅助稳定代码输出结构，防止引入奇怪的 Token。 |
| **4.7 Top K** | `40` | 限制采样候选池为 40 个最高概率 Token，增加代码输出的严谨性和可预测性。 |
| **4.8 Min P** | `0.05` | **防假死核心参数**。动态剔除概率低于最大概率 5% 的 Token，能极有效遏制模型胡说八道、突发性胡言乱语和复读假死。 |
| **4.9 重复惩罚** | `1.05` | **防死循环核心参数**。将重复惩罚从默认 1.0 微调至 1.05，可以在不破坏正常代码结构（括号、变量名重复）的前提下有效打破"无限自我复读"的死循环。 |
| **4.10 Presence Penalty** | `0.0` | 保持为 0。Presence Penalty 会降低已出现 Token 的权重，开启后可能干扰代码中必要的重复词汇，设为 0 保证代码生成的原生流畅度。 |
| **4.11 TTL（秒）** | `7200` | 维持激活状态 2 小时。Qwen 3.6 27B 作为主力开发模型需长时间保持活跃，避免因 TTL 过期频繁重新加载（每次加载约 30~60 秒），打断开发节奏。 |

### 4.12 ~ 4.14 思考与工具参数
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **4.12 启用思考** | `no` | Qwen 3.6 是标准 Coder 模型，非 R1 推理模型。开启思考只会浪费 Token 输出生成无实际价值的内部推理链，关闭后可成倍提升 Token 输出速度。 |
| **4.13 思考 Token 限制** | `no` | 配合 4.12 关闭思考一同关闭，无需设置。 |
| **4.14 限制工具结果 Token** | `no` | **⚠️ 关键避坑配置**。强烈建议设为 **`no`**。开启此选项后，Roo Code / Cline 读取的代码文件会被截断，导致 JSON/XML/Markdown 格式破损，模型解析工具返回值时出错，进而不断重试调用同一个工具，陷入"工具调用 → 截断 → 解析失败 → 重试"的无限死循环。在 64GB 内存下关闭此选项完全安全。 |

### 4.15 ~ 4.16 运行时配置
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **4.15 强制采样** | `no` | 设为 `no` 允许前端工具（Roo Code / Cline）按需动态覆盖温度和 Top P 参数，保持灵活调整空间。 |
| **4.16 Trust Remote Code** | `yes` | MLX 模型加载需要运行自定义 modeling 代码。设为 `yes` 保证模型能顺利初始化，否则可能报错无法加载。 |

### 4.17 聊天模板参数
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **enable_thinking** | `false`（不强制） | 不强制约束，交给前端工具按需控制。设置为 `false` 默认关闭思考链。 |
| **reasoning_effort** | `medium`（不强制） | 不强制约束。`medium` 在代码生成的质量和速度之间取得平衡。 |
| **自定义模板** | 暂不添加 | 如有特殊需要（如添加 `stop` 参数），可在此处按需添加键值对。键和值由您根据实际需求定义。 |

### 4.18 TurboQuant KV Cache
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **TurboQuant KV Cache** | `yes` | 开启 KV 缓存量化。在 64GB 统一内存上，多轮对话的 KV 缓存会迅速增长，量化后可大幅降低内存消耗，同时提升缓存读写速度。 |
| **Bits per channel** | `4-bit` | 4-bit 是密集模型 KV 量化的最佳平衡点：在节省约 75% KV 缓存内存的同时，能保持足够的注意力精度，不会影响代码逻辑的多轮关联能力。 |

### 4.19 SpecPrefill
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **SpecPrefill** | `no` | 已启用 DFlash 投机解码，SpecPrefill 与其功能重叠且存在冲突，必须关闭。 |

### 4.20 DFlash 块扩散投机解码（全部子项）

#### 4.20 DFlash 主开关
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **DFlash** | `yes` | **开启**。对于稠密型 27B 模型，DFlash 是性价比最高的加速手段，可带来 3~4x 的无损解码速度提升。 |

#### 4.20.1 Draft Model
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **Draft Model** | `z-lab/Qwen3.6-27B-DFlash` | ✅ **真实存在的官方 DFlash 草稿模型**。仓库地址：`https://huggingface.co/z-lab/Qwen3.6-27B-DFlash`，可直接下载使用。该草稿模型专为 Qwen 3.6 27B 稠密模型设计，保持了与主模型高度一致的生成分布，确保验证成功率最大化。 |

#### 4.20.2 Quantization（草稿模型量化）
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **Quantization** | `no` | **不开启量化**。64GB 统一内存充裕，保留 Draft 模型全精度权重可最大化 Token 匹配率（Verify Success Rate），生成速度最快。若后续内存紧张可考虑开启。 |

#### 4.20.3 Max Context
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **Max Context** | `4096` | 设置为草稿模型支持的上下文限制。当 Token 数达到或超过此阈值时，引擎自动回退到标准解码模式，防止 DFlash 在超长上下文中因草稿模型退化而导致生成卡顿或质量下降。 |

#### 4.20.4 Long-context tuning
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **Draft window size** | `1024` | 草稿模型滑动注意力窗口大小。1024 为 DFlash 默认值，可在长上下文中稳定草稿生成质量，无需修改。 |
| **Draft sink size** | `64` | 注意力汇聚 Token 数。始终保持首部 64 个 Token 的全局注意力，防止长上下文中早期信息的丢失，默认值即可。 |

#### 4.20.5 Verify mode
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **Verify mode** | `adaptive` | 自适应验证模式。引擎根据当前上下文和草稿质量自动选择最优验证策略（dflash / ddtree），无需手动干预。 |

#### 4.20.6 In-memory cache
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **In-memory cache** | `yes` | 启用 DFlash L1 前缀快照缓存，加速多轮对话中共享前缀的处理速度。 |
| **Max entries** | `8` | 缓存 8 个对话前缀快照，足够覆盖日常多轮开发对话的需求。 |
| **Size (GiB)** | `4.0` | 分配 4GB 内存作为 L1 缓存预算。当缓存超过限制时自动 LRU 淘汰最旧条目。 |

#### 4.20.7 SSD cache size
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **SSD cache size** | `no` | 64GB 统一内存充裕，无需将 L1 缓存溢出到 SSD。SSD 延迟远高于内存，开启反而可能拖慢响应速度。 |

### 4.21 Native MTP
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **Native MTP** | `no` | 当前量化版本转换时丢失了 `mtp.*` 张量，开启会导致 OMLX 报错无法启动。关闭以防止错误。具体启用方式见章节 2.3。 |

### 4.22 VLM MTP
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **VLM MTP** | `no` | 仅限 Gemma 4 视觉多模态模型（VLM）使用，Qwen 3.6 文本模型不适用，必须关闭。 |

## 4. 前端调用工具 (Roo Code / Cline) 专用系统提示词

> 以下提示词建议配置在前端工具的 System Prompt / 自定义指令中，从逻辑层面杜绝假死和工具死循环。

```markdown
# 智能体运行准则（防止假死与工具死循环）

## 1. 工具调用截断保护
- 当你读取文件或获取工具输出时，若发现内容不完整、缺少结尾括号、JSON/XML 结构错乱、或包含 [TRUNCATED] 字样，这表明工具返回结果受到了限制。
- **绝对禁止** 在不作任何修改的情况下连续重复调用同一个读取工具。
- 解决方案：必须改用分段读取命令（如指定行范围），或修改检索逻辑只读取特定区域，或直接告知用户文件过长需要人工介入。

## 2. 自我复读阻断机制
- 在任何情况下，如果你发现你的输出中包含与上一个回复高度重合的重复文本（自我复读/无限循环现象），必须立刻停止当前生成。
- 立即采取以下任意手段打破循环：① 换用完全不同的措辞重新总结问题；② 向用户提出一个明确的具体问题以打断循环；③ 调用一个不同的工具（如搜索而非读取）改变输出模式。

## 3. 工具调用失败恢复策略
- 当工具调用返回错误时，最多重试 1 次。如果再次失败，必须换用替代方案（如改用其他工具、询问用户、或输出明确的错误说明）。
- **绝对禁止** 连续 3 次以上调用同一个失败的工具而不改变策略。

## 4. 代码设计与重构标准
- **Java 开发**：严格遵循 Spring Boot 最新规范及阿里巴巴 Java 开发规约；分层清晰（Controller / Service / Repository）；异常处理禁止吞掉堆栈信息。
- **前端开发**：遵循组件化规范和单向数据流（如 React/Vue 最佳实践）；避免大型组件，合理划分职责边界。
- **小程序开发**：注意内存占用与性能瓶颈，避免一次性加载大量数据，合理使用分页和虚拟列表。
- **架构能力**：所有重构动作必须保证向下兼容；在重构核心模块前，先梳理调用链，并在代码中清晰注释核心逻辑和变更原因。
- **编码规范**：命名清晰（类名大驼峰、方法/变量小驼峰、常量全大写下划线）；工具类、接口类必须定义明确的职责边界，禁止出现上帝类（God Class）。
```

## 5. 配置总结与注意事项

| 关注点 | 配置策略 |
| :--- | :--- |
| **防假死 / 防死循环** | Min P=0.05 + 重复惩罚=1.05 + 限制工具结果Token=no + 关闭思考 |
| **防内存溢出** | Ctx Window=32K + TurboQuant KV=4-bit + DFlash Max Context=4096 降级保护 |
| **Token 输出速度** | DFlash 开启 + SpecPrefill 关闭 + 思考关闭 |
| **工具调用稳定性** | 限制工具结果Token=no（最关键！）+ 前端提示词中的失败恢复策略 |
| **模型准确性** | 温度=0.2 + 强制采样=no + Trust Remote Code=yes |

> **⚠️ 特别提醒**：如果您后续下载了带 MTP 头的 Qwen 3.6 官方版本，并成功用最新 mlx-lm 重新转换，可尝试开启 **Native MTP** 取代 DFlash，以降低内存占用。届时请同步关闭 DFlash。
