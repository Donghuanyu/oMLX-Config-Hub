# Qwen3.6-35B-A3B-UD-MLX-4bit 最佳模型设置

> **生成平台**：OMLX 0.3.10  
> **运行硬件**：Mac mini 4 M4（内存 64GB 统一内存，SSD 2TB）  
> **使用场景**：Java 后端开发 / 前端开发 / 小程序开发 / 代码重构

---

## 1. 模型基本信息

| 属性 | 值 |
| :--- | :--- |
| **模型名称** | Qwen3.6-35B-A3B-UD-MLX-4bit |
| **模型架构类型** | **混合专家模型 (MoE, Mixture of Experts)** |
| **总参数量** | 35B（全部专家权重常驻统一内存） |
| **活跃参数量** | 约 3B（A3B = Active 3B，每次 Token 仅激活 3B 参数） |
| **4bit 量化后内存占用** | 约 18~20 GB（UD 优化后内存占用略低于标准版） |
| **架构说明** | MoE（混合专家）架构的核心优势在于：总参数 35B 提供了大模型的表达深度和逻辑广度，而每次推理仅激活约 3B 参数实现极低的计算开销。这使得它在 **逻辑能力接近 35B 稠密模型** 的同时，**计算速度接近 3B 稠密模型**。但由于全部专家权重仍常驻内存，它对总内存容量的要求等同于 35B 模型级别。 |
| **UD-MLX 优化说明** | **UD（Uniform Decomposition，均匀分解）** 是 MLX 社区开发的一种高级量化优化技术。它通过将权重矩阵分解为更均匀的成分，在同样 4-bit 位宽下比标准量化保留更多模型精度。**MLX** 标识表明此版本通过官方 `mlx-lm` 转换器生成，针对 Apple Silicon（M4）的 Metal GPU 进行了指令级优化，在推理速度和兼容性上经过专门调校。 |

## 2. 推理加速技术深度解析

### 2.1 DFlash 块扩散投机解码 — 推荐：✅ **开启**

| 项目 | 说明 |
| :--- | :--- |
| **为什么需要** | 尽管 MoE 模型仅激活 3B 参数使得计算负载极低，但在本地单请求模式下，解码速度的瓶颈仍然是统一内存带宽（需要从内存读取全部 35B 权重）。DFlash 通过引入专为 35B MoE 架构设计的草稿模型，并行预测多 Token 并由主模型验证，可以显著突破内存带宽瓶颈。UD 优化版在同样参数规模下内存带宽需求略低于标准版，但 DFlash 带来的加速收益依然显著。 |
| **核心优点** | ① **3~4x 解码速度提升**，让 35B 级模型获得接近 7B 模型的 Token 输出速度；② **无损精度**，主模型验证保证代码生成的逻辑深度和正确性；③ **自动降级保护**，超长上下文自动回退标准解码。 |
| **核心缺点** | ① 需额外加载 MoE 专用的 DFlash 草稿模型（约 3~5 GB 内存），但在 64GB 内存上完全可行；② 草稿模型上下文窗口有限，超过 4096 Token 自动降级；③ 仅支持单请求流。 |
| **Draft 模型推荐** | `z-lab/Qwen3.6-35B-A3B-DFlash` — 官方提供的 DFlash 草稿模型，专为 Qwen 3.6 35B A3B MoE 架构匹配，可从 HuggingFace 直接下载。 |

### 2.2 SpecPrefill 投机预填充 — 推荐：❌ **关闭**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 用极小的草稿模型加速超长 Prompt 的预填充阶段，降低首字延迟。 |
| **为何关闭** | ① 与 DFlash 存在功能重叠且会相互冲突；② 短 Prompt 下无加速效果；③ MoE 模型本身计算预填充速度已经很快，SpecPrefill 的额外收益极有限。开启 DFlash 后此项应保持关闭。 |

### 2.3 Native MTP 原生多 Token 预测 — 推荐：❌ **关闭（当前不可用）**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 利用模型内置 MTP 预测头一次性预测多个未来 Token，无需草稿模型，约 1.5x 提速。 |
| **当前状态** | **不可用**。UD-MLX 量化版本在转换过程中同样遗失了 `mtp.*` 权重张量，OMLX 加载会报 `Config declares MTP layers but the converted weights are missing mtp.* tensors` 错误。 |
| **启用指南** | ① 下载带有 MTP 头的官方 HF 原生仓库（关注 Qwen 官方发布的带 `-MTP` 后缀版本）；② 使用最新合并 PR #990 和 PR #15 的 `mlx-lm` 重新量化转换。 |

## 3. OMLX 0.3.10 模型设置参数列表

### 4.1 模型类别
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **模型类别** | `llm` | 强制指定为纯文本大语言模型，避免自动检测误判为 VLM 或其他类别，确保工具调用稳定。 |

### 4.2 REASONING PARSER
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **REASONING PARSER** | `qwen_3_5` | Qwen 3.6 MoE 架构选用 `qwen_3_5` 解析器（根据文档说明，`qwen_3_5` 和 `qwen_3_coder` 均适用于 Qwen 3.5/3.6），可正确截取模型的推理链输出结构。 |

### 4.3 ~ 4.11 采样与输出参数
| 配置项 | 推荐设定值 | 针对防假死、防死循环与输出质量的设计理由说明 |
| :--- | :--- | :--- |
| **4.3 Ctx Window** | `131072` | **✅ 完全可行**。基于以下实测基准数据推算，64GB 统一内存在 128K 上下文下仍绰绰有余：① 基准测试中 `pp32768`（32K）Peak Mem 仅 **20.36 GB**；② KV Cache 在 4-bit TurboQuant 量化下，从 1K→32K 仅增长约 2GB（18.31→20.36 GB），呈线性增长趋势；③ 线性外推到 128K，KV Cache 预计占用约 8~10GB；④ 完整内存预算见下方节 6.1。128K 上下文彻底解决大项目多文件分析时"Context Window 不足"的报错，让 Claude Code / Roo Code 从容处理复杂调用链和跨文件重构。 |
| **4.4 最大 Token 数** | `8192` | 由于 Ctx Window 从 32K 提升至 128K，单次请求可容纳的对话历史和上下文大幅增加，模型可能需要输出更长代码（如完整类定义、多文件重构结果）。将最大 Token 数从 `4096` 提高至 `8192`，在保证 **防发散防假死** 的前提下，充分释放模型在长上下文场景中的输出能力。8192 Token 足够覆盖一次完整的类级重构输出，同时仍具备截断保护。 |
| **4.5 温度** | `0.2` | 低温度确保代码生成的高确定性，杜绝胡言乱语。 |
| **4.6 Top P** | `0.9` | 剔除长尾低概率 Token，稳定输出结构。 |
| **4.7 Top K** | `40` | 限制采样池大小，增加代码输出的严谨性。 |
| **4.8 Min P** | `0.05` | **防假死核心参数**。动态剔除概率低于最大概率 5% 的 Token，防止模型产生突发性胡言乱语或陷入复读假死。 |
| **4.9 重复惩罚** | `1.05` | **防死循环核心参数**。微调至 1.05 可强力打破"无限自我复读"的死循环，同时不破坏代码中正常的括号和变量重复。 |
| **4.10 Presence Penalty** | `0.0` | 保持默认，防止改变正常代码词汇的出现倾向，确保代码输出的原生态。 |
| **4.11 TTL（秒）** | `7200` | 维持激活状态 2 小时。作为可选主力模型，避免频繁重新加载（35B MoE 加载时间较长），保证开发连续性。 |

### 4.12 ~ 4.14 思考与工具参数
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **4.12 启用思考** | `no` | Qwen 3.6 非 R1 推理模型，关闭思考可以成倍提升 Token 输出速度，避免生成无价值的内部推理链浪费 Token 和时间。 |
| **4.13 思考 Token 限制** | `no` | 配合 4.12 关闭思考一同关闭，无需设置。 |
| **4.14 限制工具结果 Token** | `no` | **⚠️ 关键避坑配置**。强烈建议设为 **`no`**。开启后会导致 Roo Code / Cline 读取的代码文件被截断，引发工具返回的 JSON/XML 格式破损，模型解析失败后不断重试，陷入"工具调用 → 截断 → 解析失败 → 重试"的无限死循环。64GB 内存下关闭此选项安全无虞。 |

### 4.18 TurboQuant KV Cache
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **TurboQuant KV Cache** | `yes` | **在 128K 上下文下至关重要**。开启 KV 缓存量化可将 KV Cache 内存占用降低约 75%，原本 128K 需要约 32~40GB 的 KV Cache 直接压缩至 8~10GB，是支撑 128K 上下文的核心技术保障。 |
| **Bits per channel** | `4-bit` | 4-bit 是 MoE 模型 KV 量化的最佳选择，在节省约 75% KV 缓存内存的同时保持足够的注意力精度。|

### 4.20 DFlash 块扩散投机解码（全部子项）

## 4. 前端调用工具 (Roo Code / Cline) 专用系统提示词

```markdown
# 智能体运行准则（防止假死与工具死循环）

## 1. 工具调用截断保护
- 当你读取文件或获取工具输出时，若发现内容不完整、缺少结尾括号、JSON/XML 结构错乱、或包含 [TRUNCATED] 字样，这表明工具返回结果受到了限制。
- **绝对禁止** 在不作任何修改的情况下连续重复调用同一个读取工具。
- 解决方案：必须改用分段读取命令（如指定行范围），或修改检索逻辑只读取特定区域，或直接告知用户文件过长需要人工介入。

## 2. 自我复读阻断机制
- 在任何情况下，如果你发现你的输出中包含与上一个回复高度重合的重复文本（自我复读/无限循环现象），必须立刻停止当前生成。
- 立即采取以下任意手段打破循环：① 换用完全不同的措辞重新总结问题；② 向用户提出一个明确的具体问题以打断循环；③ 调用一个不同的工具（如搜索而非读取）改变输出模式。

## 3. 工具调用失败恢复策略
- 当工具调用返回错误时，最多重试 1 次。如果再次失败，必须换用替代方案（如改用其他工具、询问用户、或输出明确的错误说明）。
- **绝对禁止** 连续 3 次以上调用同一个失败的工具而不改变策略。

## 4. 代码设计与重构标准
- **Java 开发**：严格遵循 Spring Boot 最新规范及阿里巴巴 Java 开发规约；分层清晰（Controller / Service / Repository）；异常处理禁止吞掉堆栈信息。
- **前端开发**：遵循组件化规范和单向数据流（如 React/Vue 最佳实践）；避免大型组件，合理划分职责边界。
- **小程序开发**：注意内存占用与性能瓶颈，避免一次性加载大量数据，合理使用分页和虚拟列表。
- **架构能力**：所有重构动作必须保证向下兼容；在重构核心模块前，先梳理调用链，并在代码中清晰注释核心逻辑和变更原因。
- **编码规范**：命名清晰（类名大驼峰、方法/变量小驼峰、常量全大写下划线）；工具类、接口类必须定义明确的职责边界，禁止出现上帝类（God Class）。
```

## 5. Qwen3.6-35B-A3B-UD-MLX-4bit 与 Qwen3.6-35B-A3B-4bit 详细对比

| 对比维度 | Qwen3.6-35B-A3B-4bit（标准版） | Qwen3.6-35B-A3B-UD-MLX-4bit（UD优化版） |
| :--- | :--- | :--- |
| **量化方式** | 标准 MLX 4-bit 量化 | UD（Uniform Decomposition）均匀分解 4-bit 量化 |
| **精度保留** | 标准精度 | **精度更高** — UD 技术将权重矩阵分解为更均匀的成分，在相同 4-bit 位宽下保留了更多原始模型精度，输出质量略优于标准版 |
| **内存占用** | 约 20~22 GB | **约 18~20 GB（更低）** — UD 分解优化减少了量化误差导致的权重冗余，实际加载内存占用比标准版低约 2GB |
| **转换工具** | 通用 MLX 转换 | **官方 mlx-lm 转换** — 经过 Apple Silicon（M4）Metal GPU 指令级优化，推理执行效率更高 |
| **推理速度** | 标准速度 | **略快** — UD 分解后的权重矩阵在 GPU 上计算更高效，加上 MLX 的 Metal 优化，解码速度通常比标准版快 5~10% |
| **输出质量** | 良好 | **更优** — UD 将量化噪声均匀分布，减少了标准 4-bit 量化中可能出现的"异常突出"精度损失，在处理复杂代码逻辑和长上下文时表现更稳定 |
| **兼容性** | 标准兼容 | 完全兼容 — 与标准版共享相同的模型架构，OMLX 设置、DFlash 草稿模型、系统提示词完全通用 |

### 核心差异总结

> **UD-MLX 版 = 标准版 + 均匀分解优化 + MLX 框架专门适配**。本质上是同一基础模型经过更精细的量化处理和平台优化，实现了 **更低内存占用 + 更高推理精度 + 更快解码速度** 的三重提升。在 Mac mini M4 上，UD-MLX 版是更优的选择。

## 6. 配置总结与建议

### 6.1 配置速查表

| 关注点 | 配置策略 |
| :--- | :--- |
| **防假死 / 防死循环** | Min P=0.05 + 重复惩罚=1.05 + 限制工具结果Token=no + 关闭思考 |
| **防内存溢出** | Ctx Window=128K + TurboQuant KV=4-bit（必须）+ DFlash Max Context=4096 自动降级保护 |
| **Token 输出速度** | DFlash 开启 + SpecPrefill 关闭 + 思考关闭 |
| **工具调用稳定性** | 限制工具结果Token=no（最关键！）+ 前端提示词中的失败恢复策略 |
| **模型准确性** | 温度=0.2 + 强制采样=no + Trust Remote Code=yes |
| **内存预算** | 主模型 ~19GB + Draft ~4GB + KV 缓存(128K,4-bit) ~10GB + L1 缓存 4GB + 系统/其他 ~8GB = **约 45GB** ← 64GB 总内存 **完全充足，余量约 19GB** |

### 6.2 128K Ctx Window 可行性详细分析

#### ✅ 为何可以设置为 131072 (128K)

基于您在 [`Qwen3.6-35B-A3B-UD-MLX-4bit-setting.md`](0.3.10/deepseek/Qwen3.6-35B-A3B-UD-MLX-4bit-setting.md) 中已有的基准测试数据，我们做以下分析：

| 上下文长度 (Prompt) | 实测 Peak Mem | KV Cache 增量估算 |
|:---|:---|:---|
| pp1024 (1K) | **18.31 GB** | 基准（极小） |
| pp4096 (4K) | **19.42 GB** | +1.11 GB |
| pp8192 (8K) | **20.10 GB** | +0.68 GB |
| pp16384 (16K) | **21.27 GB** | +1.17 GB |
| pp32768 (32K) | **20.36 GB** | —（缓存复用导致波动） |
| **pp131072 (128K) 推算** | **≈ 28~30 GB** | **+8~10 GB（4-bit 量化后）** |

**关键推演逻辑**：
1. **KV Cache 增长规律**：从 1K→32K，Peak Mem 仅增加约 2GB（18.31→20.36 GB），说明 4-bit TurboQuant 极大压缩了 KV Cache。
2. **线性外推**：128K 是 32K 的 4 倍，KV Cache 理论上增加 4 倍 ≈ 8~10 GB（已考虑 4-bit 量化后的节省效果）。
3. **完整内存占用**：主模型 19GB + KV Cache 10GB + Draft 4GB + L1 Cache 4GB + 系统开销 8GB ≈ **45GB**，64GB 总内存 **剩余约 19GB 安全余量**。

#### ⚠️ 需要注意的性能折衷

| 关注点 | 32K 配置 | 128K 配置 | 说明 |
|:---|:---|:---|:---|
| **首字延迟 (TTFT)** | ≈ 74 秒（32K 预填充） | ≈ **150~300 秒**（128K 预填充） | 128K 预填充时间显著增长。**预填充仅在对话开始时发生一次**，后续工具调用只生成输出 Token 无需重新预填充。建议首次加载时耐心等待。 |
| **DFlash 加速覆盖** | 短 Prompt 下 3~4x 加速 | 短 Prompt 下同样 3~4x 加速 | DFlash Max Context=4096 不变，短上下文场景加速不受影响。长上下文自动降级到 MoE 原生解码（33+ tok/s）。 |
| **生成速度 (长上下文)** | 33.5 tok/s（32K 降级后） | **33.5 tok/s**（同 32K，因 DFlash 降级后速度取决于 MoE 解码） | MoE 模型的 33+ tok/s 原生解码速度对于代码生成依然流畅可接受。 |
| **多模型共存** | 可同时激活 3 个模型 | **建议仅激活 2 个模型**（gemma-4 + 本模型） | 128K 下 KV Cache 占用增多，建议减少同时激活的模型数量以确保内存余量。 |

#### 💡 最佳实践建议

1. **首次加载等待**：OMLX 激活模型并预填充 128K 上下文的首次 TTFT 可能达到 2~5 分钟，这是正常现象。预填充完成后，后续工具调用的响应速度将恢复正常。
2. **分段读取策略**：虽然 Ctx Window 设为 128K，仍建议前端工具（Roo Code / Cline）采用分段读取文件策略，避免单次请求耗尽 Token 预算导致意外截断。
3. **若 TTFT 不可接受**：如果 128K 的 TTFT 在实际使用中过长，您可以退而使用 **65536 (64K)** 作为折中方案，此时 TTFT 约 150 秒，内存占用约 23~25GB。
4. **开启 SpecPrefill 的替代方案**：如果 128K 下 TTFT 过长无法忍受，可考虑关闭 DFlash，开启 **SpecPrefill**（选择本地一个小模型如 `qwen-1.5b` 作为草稿模型），将预填充加速约 2~3 倍，大幅降低 TTFT。

## 7. 前端调用工具上下文窗口设置指南

在使用 128K Ctx Window 的 OMLX 后端模型时，前端工具（Claude Code / Roo Code）的上下文窗口设置必须同步调整，否则仍然可能出现"上下文不足"的报错。

### 7.1 Claude Code 设置（`settings.json`）

Claude Code 的配置位于 [`.claude/settings.json`](.claude/settings.json)。以下是针对 128K 上下文推荐的设置：

```json
{
  "env": {
    "ANTHROPIC_BASE_URL": "http://xxx.xxx.xxx:8000",
    "ANTHROPIC_AUTH_TOKEN": "XXX",
    "ANTHROPIC_DEFAULT_OPUS_MODEL": "Qwen3.6-35B-A3B-UD-MLX-4bit",
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "Qwen3.6-35B-A3B-UD-MLX-4bit",
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "Qwen3.6-35B-A3B-UD-MLX-4bit",
    "API_TIMEOUT_MS": "3000000",
    "CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC": "1",
    "CLAUDE_CODE_ATTRIBUTION_HEADER": "0",
    "CLAUDE_MAX_CONTEXT_TOKENS": "131072",
    "CLAUDE_SUMMARY_THRESHOLD": "98304",
    "CLAUDE_CODE_MAX_CONCURRENT_REQUESTS": "1",
    "CLAUDE_CODE_OVERRIDE_MODEL_MAX_TOKENS": "8192"
  }
}
```

#### 各配置项说明

| 环境变量 | 当前值 | 推荐值 | 作用说明 |
|:---|:---:|:---:|:---|
| `CLAUDE_MAX_CONTEXT_TOKENS` | `49152` | `131072` | **核心参数**。Claude Code 最大上下文 Token 数。必须与 OMLX 后端的 Ctx Window (131072) 匹配，否则即使后端支持 128K，前端也会在达到 49152 时强行截断上下文。 |
| `CLAUDE_SUMMARY_THRESHOLD` | `32768` | `98304` | 上下文摘要压缩阈值。当上下文 Token 数超过此值时，Claude Code 开始对早期对话进行摘要压缩。设为 98304（128K 的 75%）可在接近极限时才开始压缩，保持上下文完整性。 |
| `CLAUDE_CODE_OVERRIDE_MODEL_MAX_TOKENS` | 未设置 | `8192` | 覆盖模型的最大输出 Token 数。与 OMLX 的"最大 Token 数 8192"保持一致，确保模型输出的长度不被前端额外截断。 |
| `API_TIMEOUT_MS` | `3000000` | `3000000` | API 超时时间（毫秒）。**建议保持不变**。300 秒的超时在 128K 下仍然足够，因为 TTFT（首字延迟）虽然可能达到 150~300 秒，但超时计时器通常从请求发起时开始计算，需要确保覆盖 128K 的完整预填充时间。如果实际使用中遇到超时，可考虑上调至 `6000000`（600 秒）。 |

### 7.2 Roo Code / Cline 设置
