# gemma-4-26b-a4b-it-4bit 最佳模型设置

> **生成平台**：OMLX 0.3.10  
> **运行硬件**：Mac mini 4 M4（内存 64GB 统一内存，SSD 2TB）  
> **使用场景**：Java 后端开发 / 前端开发 / 小程序开发 / 代码重构  
> **特殊角色**：固定激活常驻模型

---

## 1. 模型基本信息

| 属性 | 值 |
| :--- | :--- |
| **模型名称** | gemma-4-26b-a4b-it-4bit |
| **模型架构类型** | **混合专家模型 (MoE, Mixture of Experts)** |
| **总参数量** | 26B（全部专家权重常驻统一内存） |
| **活跃参数量** | 约 4B（a4b = Active 4B，每次 Token 仅激活 4B 参数） |
| **4bit 量化后内存占用** | 约 15~17 GB |
| **架构说明** | 由 Google 开发的 MoE 模型。总参数量 26B 提供了深厚的知识储备，但每次推理仅激活约 4B 参数，计算开销极低。这使得它在 Apple Silicon 上能达到类似 4B 稠密模型的响应速度，却拥有 26B 级别的逻辑综合能力。4bit 量化后仅占用约 16GB，作为系统中"固定激活"的常驻模型非常理想，可与 Qwen 系列模型并行使用而不超出 64GB 内存上限。 |

## 2. 推理加速技术深度解析

### 2.1 DFlash 块扩散投机解码 — 推荐：✅ **开启**

| 项目 | 说明 |
| :--- | :--- |
| **为什么需要** | 作为固定激活的主力常驻模型，虽然每次计算仅需激活 4B 参数，但瓶颈在于从统一内存读取全部 26B 权重的带宽需求。DFlash 通过专为 Gemma 4 设计的草稿模型进行并发扩散预测，可以极大缓解内存带宽瓶颈，实现 3~4x 的无损解码加速。 |
| **核心优点** | ① **3~4x 端到端解码速度提升**，让 26B 模型的响应速度接近 3~7B 级模型；② **无损精度**，主模型验证确保输出与直接生成完全一致；③ **自动降级保护**，超长上下文自动回退标准解码。 |
| **核心缺点** | ① 需额外加载 `z-lab/gemma-4-26B-A4B-it-DFlash` 草稿模型（约 3~5 GB），64GB 内存可承受；② 草稿模型上下文窗口有限，超过阈值后自动降级；③ 仅支持单请求流。 |
| **Draft 模型推荐** | `z-lab/gemma-4-26B-A4B-it-DFlash` — Google 官方提供的 DFlash 草稿模型，可直接从 HuggingFace 下载。 |

### 2.2 SpecPrefill 投机预填充 — 推荐：❌ **关闭**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 用小型草稿模型加速超长 Prompt 的预填充阶段，降低首字延迟。 |
| **为何关闭** | ① 与 DFlash 存在功能重叠且冲突；② 短 Prompt 下有额外开销无收益；③ Gemma 4 MoE 的预填充计算已足够快。DFlash 开启时此项必须关闭。 |

### 2.3 Native MTP 原生多 Token 预测 — 推荐：❌ **关闭（架构不兼容）**

| 项目 | 说明 |
| :--- | :--- |
| **为何关闭** | Gemma 4 系列模型架构本身**不支持** Qwen 系列那样的原生 MTP 架构。此加速功能在 Gemma 4 的架构层面完全不适用，必须设为 `no`。不存在"下载正确版本即可使用"的情况。 |

## 3. OMLX 0.3.10 模型设置参数列表

### 4.1 模型类别
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **模型类别** | `llm` | 强制指定为纯文本大语言模型，避免自动检测误判为 VLM（Gemma 4 也有 VLM 版本）或其他类别。 |

### 4.2 REASONING PARSER
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **REASONING PARSER** | `None` | Gemma 4 不是专门的推理模型（非 R1 系列），也不支持 Qwen 或 DeepSeek 的专用 Reasoning Parser。设为 `None` 可避免不必要的解析错误，让模型以原生格式输出。 |

### 4.3 ~ 4.11 采样与输出参数
| 配置项 | 推荐设定值 | 针对防假死、防死循环与输出质量的设计理由说明 |
| :--- | :--- | :--- |
| **4.3 Ctx Window** | `32768` | 64GB 内存充裕。32K Token 上下文可容纳完整的大型项目代码结构和多文件上下文。Gemma 4 的 MoE 架构在长上下文中仍能保持较低计算开销。注意：当上下文超过 DFlash 草稿模型的 4096 限制时，引擎会自动降级到标准解码，仍可正常处理 32K 上下文。 |
| **4.4 最大 Token 数** | `4096` | 限制单次最大输出长度，防止模型发散导致无限输出或假死。 |
| **4.5 温度** | `0.2` | 黄金低温度，确保代码生成高确定性和严谨性，杜绝胡言乱语。 |
| **4.6 Top P** | `0.9` | 剔除长尾低概率 Token，在低温度下辅助稳定代码输出结构。 |
| **4.7 Top K** | `40` | 限制采样池大小，增加代码输出的严谨性和可预测性。 |
| **4.8 Min P** | `0.05` | **防假死核心参数**。动态剔除概率低于最大概率 5% 的 Token，可极有效遏制模型胡说八道和复读假死。 |
| **4.9 重复惩罚** | `1.05` | **防死循环核心参数**。微调至 1.05 可强力打破"无限自我复读"的死循环，同时不破坏代码中正常的括号和变量重复结构。 |
| **4.10 Presence Penalty** | `0.0` | 保持默认，防止改变正常代码词汇的出现倾向。 |
| **4.11 TTL（秒）** | `7200` | 维持激活状态 2 小时。Gemma 4 作为固定激活常驻模型，需要长时间保持活跃，避免频繁重新加载。 |

### 4.12 ~ 4.14 思考与工具参数
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **4.12 启用思考** | `no` | Gemma 4 it（instruction-tuned）版本非 R1 推理模型，关闭思考可成倍提升 Token 输出速度，避免无效思考浪费 Token。 |
| **4.13 思考 Token 限制** | `no` | 配合 4.12 关闭思考一同关闭。 |
| **4.14 限制工具结果 Token** | `no` | **⚠️ 关键避坑配置**。强烈建议设为 **`no`**。开启后会导致读取代码文件被截断，JSON/XML 格式破损，模型解析失败后不断重试，陷入"工具调用 → 截断 → 解析失败 → 重试"的无限死循环。64GB 内存下关闭此选项安全无虞。 |

### 4.15 ~ 4.16 运行时配置
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **4.15 强制采样** | `no` | 允许前端工具按需动态覆盖温度和 Top P 参数，保持灵活调整空间。 |
| **4.16 Trust Remote Code** | `yes` | Gemma 4 的 MLX 模型需要运行自定义 modeling 代码，设为 `yes` 保证模型加载成功。 |

### 4.17 聊天模板参数
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **enable_thinking** | `false`（不强制） | 不强制约束，默认关闭思考链。 |
| **reasoning_effort** | `medium`（不强制） | 不强制约束，留给前端工具调整空间。 |
| **自定义模板** | 暂不添加 | 如需特殊参数可按需添加键值对。 |

### 4.18 TurboQuant KV Cache
| 配置项 | 推荐设定值 | 设计理由 |
| :--- | :--- | :--- |
| **TurboQuant KV Cache** | `yes` | 开启 KV 缓存量化，在多轮对话中大幅降低 KV 缓存内存消耗。 |
| **Bits per channel** | `8-bit` | Gemma 4 对注意力精度要求较高（Google 设计特性），推荐使用 8-bit 量化。虽然节省内存幅度不如 4-bit，但能最大程度保持 Gemma 4 精细推理和多轮上下文关联的注意力精度。在 64GB 内存下，8-bit 的额外内存开销完全可接受。 |

### 4.19 SpecPrefill

## 4. 前端调用工具 (Roo Code / Cline) 专用系统提示词

```markdown
# 智能体运行准则（防止假死与工具死循环）

## 1. 工具调用截断保护
- 当你读取文件或获取工具输出时，若发现内容不完整、缺少结尾括号、JSON/XML 结构错乱、或包含 [TRUNCATED] 字样，这表明工具返回结果受到了限制。
- **绝对禁止** 在不作任何修改的情况下连续重复调用同一个读取工具。
- 解决方案：必须改用分段读取命令（如指定行范围），或修改检索逻辑只读取特定区域，或直接告知用户文件过长需要人工介入。

## 2. 自我复读阻断机制
- 在任何情况下，如果你发现你的输出中包含与上一个回复高度重合的重复文本（自我复读/无限循环现象），必须立刻停止当前生成。
- 立即采取以下任意手段打破循环：① 换用完全不同的措辞重新总结问题；② 向用户提出一个明确的具体问题以打断循环；③ 调用一个不同的工具（如搜索而非读取）改变输出模式。

## 3. 工具调用失败恢复策略
- 当工具调用返回错误时，最多重试 1 次。如果再次失败，必须换用替代方案（如改用其他工具、询问用户、或输出明确的错误说明）。
- **绝对禁止** 连续 3 次以上调用同一个失败的工具而不改变策略。

## 4. 代码设计与重构标准
- **Java 开发**：严格遵循 Spring Boot 最新规范及阿里巴巴 Java 开发规约；分层清晰（Controller / Service / Repository）；异常处理禁止吞掉堆栈信息。
- **前端开发**：遵循组件化规范和单向数据流（如 React/Vue 最佳实践）；避免大型组件，合理划分职责边界。
- **小程序开发**：注意内存占用与性能瓶颈，避免一次性加载大量数据，合理使用分页和虚拟列表。
- **架构能力**：所有重构动作必须保证向下兼容；在重构核心模块前，先梳理调用链，并在代码中清晰注释核心逻辑和变更原因。
- **编码规范**：命名清晰（类名大驼峰、方法/变量小驼峰、常量全大写下划线）；工具类、接口类必须定义明确的职责边界，禁止出现上帝类（God Class）。
```

## 5. 配置总结与注意事项

| 关注点 | 配置策略 |
| :--- | :--- |
| **防假死 / 防死循环** | Min P=0.05 + 重复惩罚=1.05 + 限制工具结果Token=no + 关闭思考 |
| **防内存溢出** | Ctx Window=32K + TurboQuant KV=8-bit + DFlash Max Context=4096 降级保护 |
| **Token 输出速度** | DFlash 开启 + SpecPrefill 关闭 + 思考关闭 |
| **工具调用稳定性** | 限制工具结果Token=no（最关键！）+ 前端提示词中的失败恢复策略 |
| **模型准确性** | 温度=0.2 + 强制采样=no + Trust Remote Code=yes |
| **内存预算** | 主模型 ~16GB + Draft ~3GB + KV 缓存 ~2GB + L1 缓存 4GB = 约 25GB，64GB 总内存非常充裕 |

> **💡 关于 Gemma 4 作为常驻模型的建议**：Gemma 4 26B（4bit 量化约 16GB）是系统中内存占用最小的模型。作为"固定激活"常驻模型，它适合承担快速响应、常规代码补全和简单问题解答等轻量级任务。当遇到复杂的架构设计或大规模重构时，可以切换到 Qwen 3.6 27B 或 35B 作为主力。
