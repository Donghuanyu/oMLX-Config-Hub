# Qwen3.6-27B-UD-MLX-4bit 最佳模型设置

## 1. 模型基本信息
* **模型名称**：Qwen3.6-27B-UD-MLX-4bit
* **模型架构类型**：**稠密型模型 (Dense)**
* **架构说明**：
  这是一款传统的全参数稠密模型，每次前向传播（Token生成）都会激活全部的 27B 参数。与 MoE（混合专家模型）相比，稠密模型在单次生成时需要进行更大量的矩阵乘法计算。在 Apple Silicon 统一内存架构下，稠密模型对内存带宽的压力是恒定且庞大的。因此，这类型模型极其依赖各类推理加速技术来提高每秒输出的 Token 数。

---

## 2. 推理加速技术深度解析

为了压榨 Mac mini 4 (64G) 的硬件潜力，此配置针对多种加速方案进行了定制：

### 2.1 DFlash 块扩散投机解码 (当前状态：启用)
* **为什么需要**：在本地单人开发场景下，大模型的生成速度（解码阶段）主要受限于内存带宽瓶颈（Memory Bandwidth Bound）。DFlash 通过引入一个小型的“草稿模型”（Draft Model）来一次性预测多个候选 Token，然后由主模型在一轮前向传播中并发验证，从而打破了自回归解码一次只能读写一次内存的限制。
* **优点**：
  * **超强加速**：提供无损的 3x 至 4x 的生成速度（TPS）提升。
  * **无损精度**：大模型的验证机制确保了输出的 Token 序列在数学上与直接用 27B 稠密模型生成的结果 100% 一致。
* **缺点**：
  * **额外内存开销**：必须常驻内存加载一个小型的 Draft 模型（通常需多占用几GB内存），不过在 64G 内存上完全够用。
  * **上下文长度限制**：当上下文超过 DFlash 草稿模型所支持的范围（通常为 4096 Token）时，推理引擎会自动降级回退到标准解码模式。
  * **并发受限**：仅支持单请求流，适合单人开发，不适合高并发多用户环境。

### 2.2 SpecPrefill 投机预填充 (当前状态：禁用)
* **原理**：用一个极小的模型快速计算超长 Prompt 的预填充阶段（Prefill），然后将状态传递给主模型验证，旨在降低首字输出延迟（TTFT）。
* **优缺点分析**：
  * **优点**：对于极长的代码文件读取（例如数万 Token），能有效降低开始输出前的“假死”等待感。
  * **缺点**：在短 Prompts 场景下没有加速效果，甚至因为小模型推理的额外消耗导致变慢。此外，它会和 DFlash 冲突，并抢占宝贵的 KV 缓存，在 64G 统一内存下开启 DFlash 已能实现极致的综合体验，因此无需开启 SpecPrefill。

### 2.3 Native MTP 原生多Token预测 (当前状态：禁用，无法开启)
* **为什么无法开启**：您当前下载的 4bit 量化版本在转换过程中遗失了原模型声明的 `mtp.*` 权重张量。开启此项会导致 OMLX 报错无法启动。
* **优点**：不需要额外加载任何草稿模型，内存开销极低，可以直接通过主模型中内置的预测头实现约 1.5x 的稳定提速。
* **缺点**：模型文件转换要求高。
* **MTP 启用指南**：
  若想使用 Native MTP 提速，您需要：
  1. 下载官方完整未裁剪的带 MTP 头的 Qwen 官方 HF 原生仓库（例如 `Qwen/Qwen2.5-Math-7B-Instruct-MTP` 类型的 MTP 原生模型仓库，若 Qwen 3.6 发布了 MTP 版本，请认准带有 `MTP` 后缀的版本）。
  2. 使用最新版已合并 PR #990 和 PR #15 补丁的 `mlx-lm` 命令行重新进行 4bit 量化转换。命令如下：
     ```bash
     python -m mlx_lm.convert --hf-path <官方包含MTP权重的HF仓库路径> -q --q-bits 4 --mlx-path <输出路径>
     ```
     这能确保转换生成的 `.safetensors` 文件中包含 `mtp.layers.*` 相关张量，从而可被 OMLX 顺利识别。

## 3. OMLX 0.3.10 模型设置参数列表

| 配置项 | 推荐设定值 | 针对防假死、防死循环与输出质量的设计理由说明 |
| :--- | :--- | :--- |
| **4.1 模型类别** | `llm` | 强制指定为文本模型，防止自动检测偶尔误判为其他类别。 |
| **4.2 REASONING PARSER** | `qwen_3_coder` | 专为 Qwen 3.6 适配的推理链解析器，确保模型逻辑输出的结构被正确截取。 |
| **4.3 Ctx Window** | `32768` | 64G 内存足够宽裕，32K 上下文能完美容纳大型项目和多个文件的上下文。 |
| **4.4 最大Token数** | `4096` | 限制单次最大输出，防范模型因发散导致无限输出。 |
| **4.5 温度** | `0.2` | 黄金低温度。确保代码生成高确定性、符合严格规范，杜绝胡言乱语。 |
| **4.6 Top P** | `0.9` | 剔除长尾低概率词，在低温度下辅助稳定代码输出结构。 |
| **4.7 Top K** | `40` | 限制采样池大小，增加代码输出的严谨性。 |
| **4.8 Min P** | `0.05` | 动态剔除概率极低的“乱语” Token，能非常有效地遏制模型胡说八道和假死复读。 |
| **4.9 重复惩罚** | `1.05` | 极其重要！微调此项（默认1.0）可强力打破“无限自我复读”的死循环，又不会破坏代码中正常的括号和变量重复。 |
| **4.10 Presence Penalty** | `0.0` | 保持为0，防止改变正常代码词汇的出现倾向。 |
| **4.11 TTL（秒）** | `7200` | 维持激活状态2小时，作为主力开发模型，避免频繁重新加载带来的等待。 |
| **4.12 启用思考** | `no` | 该模型为标准 Coder 模型非专门的 R1 推理模型，关闭思考能成倍提升 Token 输出速度，避免无效思考。 |
| **4.13 思考Token限制** | `no` | 配合 4.12 关闭。 |
| **4.14 限制工具结果Token** | `no` | **避坑核心配置**：强烈建议设为 **`no`**。开启此限制会导致 Roo Code/Cline 读取长代码时输出被截断，JSON/XML 破损引发模型解析错误，从而陷入“工具调用->被截断->失败重试”的死循环。64G 内存下关闭此限制安全无虞。 |
| **4.15 强制采样** | `no` | 允许前端调用工具按需动态覆盖温度和 Top P 参数。 |
| **4.16 Trust Remote Code** | `yes` | 允许加载 MLX 模型所需的自定义 modeling 代码，保证模型能顺利初始化。 |
| **4.17.1 聊天模板参数** | `enable_thinking` = `false` (不强制)<br>`reasoning_effort` = `medium` (不强制) | 模板参数不作强行约束，给前端工具留出调参自由。 |
| **4.18 TurboQuant KV Cache**| `yes` | 开启 KV 缓存量化，在 64G 内存上大幅节省内存并提升多轮对话处理速度。 |
| **4.18.1 Bits per channel** | `4-bit` | 4-bit 量化 KV 既能缩减内存，又能保证代码逻辑注意力的准确性。 |
| **4.19 SpecPrefill** | `no` | 已启用性能更强大的 DFlash 投机解码，此项关闭防冲突。 |
| **4.20 DFlash** | `yes` | 开启投机解码，将生成速度拉满。 |
| **4.20.1 Draft Model** | `z-lab/Qwen3.6-27B-DFlash` | **真实存在的 DFlash 草稿模型**，可直接在 HuggingFace 下载使用。 |
| **4.20.2 Quantization** | `no` | **不开启量化**。在 64G 统一内存下，保留 Draft 模型精度不量化能得到最高匹配率（Verify Success Rate），生成速度最快。 |
| **4.20.3 Max Context** | `4096` | 设置 4096 为限，超过后回退普通引擎，防止 DFlash 在超长上下文中因退化而导致生成卡顿。 |
| **4.20.4.1 Draft window size**| `1024` | 默认滑动注意力窗口，稳定长文本表现。 |
| **4.20.4.2 Draft sink size** | `64` | 维持首部 64 个 Token 保证全局注意力稳定。 |
| **4.20.5 Verify mode** | `adaptive` | 默认自适应验证，确保最优验证效率。 |
| **4.20.6 In-memory cache** | `yes` | 启用 RAM 缓存来存储聊天前缀快照，加快多轮对话的响应。 |
| **4.20.6.1 Max entries** | `8` | 缓存 8 个多轮对话的前缀。 |
| **4.20.6.2 Size (GiB)** | `4.0` | 分配 4GB 内存作为快速前缀缓存，防二次计算。 |
| **4.20.7 SSD cache size** | `no` | 拥有 64G 内存，无需将缓存溢出到 SSD，保证高带宽响应。 |
| **4.21 Native MTP** | `no` | 当前量化版缺少 mtp 张量，关闭以防止报错。 |
| **4.22 VLM MTP** | `no` | 仅限 Gemma 4 视觉模型使用，此项关闭。 |

## 4. 前端调用工具 (Roo Code / Cline) 专用系统提示词 (System Prompt)

请在前端工具的系统提示词配置中加入以下内容，从逻辑上杜绝假死和工具死循环：

```markdown
# 智能体运行准则（防止假死与工具死循环）

1. **工具调用截断保护**：
   - 当你读取文件或获取工具输出时，若发现内容不完整、缺少结尾括号或包含 [TRUNCATED] 字样，这表明工具返回结果受到了限制或文件过大。
   - **绝对禁止** 在不作修改的情况下连续重复调用同一个读取工具。
   - 解决方案：必须改用分段读取命令，或修改检索逻辑只读取特定行范围（如使用更具体的阅读或搜索工具），或者直接提醒用户由于文件过长需要由其精简或人工介入。

2. **自我复读阻断机制**：
   - 在任何情况下，如果你发现你的输出中包含与上一个回复高度重合的重复文本（自我复读现象），必须立刻停止生成，换用完全不同的措辞重新总结，或者向用户提出明确的问题以打断循环。

3. **代码设计与重构标准 (Java, 前端, 小程序)**：
   - **架构能力**：在进行 Java 开发时，严格遵循 Spring Boot 最新规范及阿里巴巴开发规约；前端开发需遵循组件化规范和单向数据流；小程序开发中必须注意内存占用与性能瓶颈。
   - **代码重构**：所有重构动作必须保证向下兼容。在重构核心模块前，先梳理调用链，并在代码中清晰注释核心逻辑。
   - **编码规范**：命名清晰（遵循驼峰或蛇形规范），异常处理禁止吞掉堆栈，所有工具类、接口类必须定义明确的职责边界。
```
