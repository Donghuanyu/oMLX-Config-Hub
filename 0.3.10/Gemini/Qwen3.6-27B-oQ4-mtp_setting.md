# Qwen3.6-27B-oQ4-mtp 最佳模型设置

> **生成平台**：OMLX 0.3.10  
> **运行硬件**：Mac mini 4 M4（内存 64GB 统一内存，SSD 2TB）  
> **使用场景**：Java 后端开发 / 前端开发 / 小程序开发 / 代码重构

---

## 1. 模型基本信息

| 属性 | 值 |
| :--- | :--- |
| **模型名称** | Qwen3.6-27B-oQ4-mtp |
| **模型架构类型** | **稠密型模型 (Dense)** |
| **参数量** | 27B（全部参数每次前向传播均被激活） |
| **oQ4 量化后内存占用** | 约 17~18 GB |
| **架构说明** | 传统的全参数稠密模型。该版本为 `oQ4-mtp`，其中 `oQ4` 表示优化后的 4-bit 量化，`mtp` 表示包含了原生多 Token 预测（Multi-Token Prediction）权重。每次前向传播仍会激活全部 27B 参数，但由于原生内置了 MTP 模块，可以在不外挂草稿模型的情况下实现显著的解码加速。 |

---

## 2. 推理加速技术深度解析

### 2.1 Native MTP 原生多 Token 预测 — 推荐：✅ **开启**

| 项目 | 说明 |
| :--- | :--- |
| **为什么需要** | 本模型是原生包含 MTP 权重的版本。Native MTP 利用模型内部训练好的 MTP 预测头（MTP heads）在一次前向传播中并发预测多个未来的 Token，相比标准单 Token 解码，能将生成速度稳定提升约 1.5 倍。 |
| **核心优点** | ① **零额外内存开销**：不需要像 DFlash 那样额外加载独立的草稿模型（Draft Model），极大节省统一内存；② **稳定性极高**：因为 MTP 权重与主模型完全一体化训练，推理匹配成功率高且稳定；③ **无冲突加载**：直接通过引擎原生驱动，不会造成内存分片或 OOM。 |
| **核心缺点** | ① 提速幅度（约 1.5x）略低于理想状态下的 DFlash（约 3x）；② 仅支持单请求流。 |

### 2.2 DFlash 块扩散投机解码 — 推荐：❌ **关闭（可选开启）**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 引入额外的草稿模型（如 `z-lab/Qwen3.6-27B-DFlash`）进行投机验证，提供 3~4x 的加速。 |
| **为何关闭** | 由于本模型已启用 **Native MTP** 提速，为了保证系统极简、不占用额外统一内存以及避免投机解码冲突，建议关闭 DFlash。如果为了压榨极致的生成速度（且不在意多占用 3~5GB 内存加载草稿模型），可以关闭 Native MTP 并开启 DFlash，配置 DFlash 草稿模型为 `z-lab/Qwen3.6-27B-DFlash`。 |

### 2.3 SpecPrefill 投机预填充 — 推荐：❌ **关闭**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 用极小的模型快速计算超长 Prompt 的预填充阶段，降低首字延迟（TTFT）。 |
| **为何关闭** | SpecPrefill 会与 Native MTP/DFlash 机制冲突，且在短 Prompt 下无加速效果，增加额外开销。对于 64GB 内存的 Mac mini 4 而言，关闭以换取最纯净和稳定的运行环境。 |

## 7. 基准测试结果深度分析与优化方案

### 7.1 基准测试数据分析

根据上述在 OMLX 0.3.10 平台上的实测数据，我们可以得出以下关键诊断：

1. **Native MTP 卓越的解码稳定性**：
   * **现象**：在 `pp1024`（1K 上下文）时，生成速度为 **`19.5 tok/s`**；在 `pp16384`（16K 上下文）下，生成速度依然能维持在 **`15.9 tok/s`**。
   * **分析**：与外挂 DFlash 草稿模型时长上下文性能崩溃（在 32K 降至 5.5 tok/s）不同，Native MTP 显示出了极其平滑的速度衰减曲线。因为 MTP 预测头是直接集成在 27B 主模型内部的，不存在草稿模型匹配率断崖式下跌 and 验证开销反噬的问题，因此在长上下文中更加可靠和稳定。
   
2. **极低且安全的内存占用**：
   * **现象**：即使在 16K 上下文的最高负荷下，峰值内存占用也仅为 **`20.69 GB`**。
   * **分析**：相较于启用 DFlash 解码在 16K 下高达 32.08 GB 的内存开销，Native MTP 节省了近 12GB 的统一内存。这为 Mac mini 4 (64GB) 留出了高达 **43GB+ 的安全余量**，完全杜绝了开发过程中多工具并发、编译运行或前端热更新导致的内存溢出（OOM）隐患。

3. **预填充（Prefill）首字延迟硬伤**：
   * **现象**：`pp4096` 时 TTFT 为 32.5s，`pp16384` 时首字延迟高达 **142.6s**。
   * **分析**：由于关闭了投机预填充（SpecPrefill），长文本的预填充阶段依然需要 27B 稠密模型进行全量矩阵运算。虽然解码流畅，但首字启动依然存在假死感。

---

### 6.2 性能优化与加速模式选择方案

基于上述分析，我们为您提供两种极佳的调优和加速方案，您可以根据开发偏好在 OMLX 中进行切换：

#### 方案 A：原生 MTP 极简稳定流（默认推荐）
* **配置方式**：保持 `Native MTP = yes`，`DFlash = no`，`SpecPrefill = no`。
* **适用场景**：经常需要多轮长对话、频繁读取中大型代码库、且背景运行其他重型应用（如多个 IDE、Docker、本地编译）的开发场景。
* **特点**：速度平稳维持在 16~20 TPS，内存仅需 20GB，稳定性绝对第一，绝不假死。

#### 方案 B：DFlash 极致爆发流（可选切换）
* **配置方式**：将 `Native MTP` 设为 **`no`**，并按下表配置 DFlash 与投机预填充：

| 配置项 | 设定值 | 优化后预期效果与收益说明 |
| :--- | :--- | :--- |
| **4.19 SpecPrefill** | **`yes`** | 启用投机预填充，结合阈值，专门降低 16K 以上 Prompt 带来的 TTFT 首字假死感。 |
| **4.19.2 Keep Rate** | **`20% — Balanced`** | 预填充阶段提速约 3 倍，使得 16K 启动时间由 142s 降至约 50s 左右。 |
| **4.19.3 Threshold** | **`4096`** | 设为 4096，保证短对话不被降采样，长文件读取自动触发提速。 |
| **4.20 DFlash** | **`yes`** | 启用外挂投机解码，追求极致的生成效率。 |
| **4.20.1 Draft Model** | **`z-lab/Qwen3.6-27B-DFlash`** | 指定官方专属的 DFlash 草稿模型，需额外下载。 |
| **4.20.3 Max Context** | **`16384`** | 设置上限为 16K，在 16K 以内享受到 **32~41 TPS** 的暴涨解码速度，超出 16K 优雅回退到原生自回归。 |

> **总结建议**：本 `oQ4-mtp` 模型相比之前的普通 4bit 版，最大的优势就在于原生 MTP 带来了“**低功耗、免额外内存、输出平滑**”的加速能力。如果您的日常开发对话长度在 16K 以内，且不想承受额外的 DFlash 草稿模型加载延迟，请坚守 **方案 A (Native MTP=yes)**，这能带来最具性价比的极简开发体验。

## 8. 更新后的基准测试评估与二次优化方案

### 8.1 更新成效深度分析

通过将测试窗口延伸至 **32K 上下文**，本次测试带来了两个极具价值的性能洞察：

1. **Native MTP 对长文本生成速度的降维打击**：
   * **现象**：在 `pp32768`（32K）超长上下文下，Native MTP 依然稳定输出 **`13.5 tok/s`**。
   * **对比**：普通 4bit 稠密模型在开启 DFlash 时，32K 解码速度会跌至 **`5.5 tok/s`**（因匹配率降低而反噬性能）；即使关闭 DFlash 回退普通解码也只有 **`7.8 tok/s`**。
   * **结论**：在 32K 上下文下，Native MTP 相比 DFlash 提速 **`145%`**，相比普通解码提速 **`73%`**。这证明了内置 MTP 结构在长对话、大代码库上下文下的输出表现完全优于投机草稿模型方案，解决了长文本下“越聊越卡”的核心痛点。

2. **绝对的内存溢出（OOM）免疫**：
   * **现象**：32K 极限压力下，峰值内存仅占用 **`23.69 GB`**（DFlash 方案在 32K 下则需 32.15 GB）。
   * **结论**：节省了 **`8.46 GB`** 统一内存。在 Mac mini 4 64GB 上拥有极高安全系数，可以完全放心开启 32K 窗口，绝不会因为多轮对话而卡死系统。

### 8.2 针对首字延迟（TTFT）的二次深度优化

虽然解码（tg TPS）极其优秀，但 32K 下 **`303.2 秒（约 5 分钟）`** 的首字延迟（TTFT）仍会造成严重的开发“假死”体验。

为了在 **不外挂 DFlash 草稿模型（保留低内存占用与高稳定度）** 的前提下解决该问题，建议在 OMLX 控制台中进行以下**二次优化微调**：

#### 🌟 终极优化：Native MTP + SpecPrefill 混合双驱模式

由于投机预填充（SpecPrefill）作用于 Prefill 阶段，而 Native MTP 作用于 Generation 阶段，二者在 OMLX 下可以完美兼容共存。配置如下：

| 配置项 | 推荐优化值 | 优化后预期收益与设计理由 |
| :--- | :--- | :--- |
| **4.19 SpecPrefill** | **`yes`** | **开启投机预填充**，利用小草稿模型专门加速首字预计算。 |
| **4.19.2 Keep Rate** | **`20% — Balanced`** | 推荐 20% 平衡比率，将 32K 的 prefill 速度提升近 3 倍。 |
| **4.19.3 Threshold** | **`4096`** | 设为 4096 Token。短 Prompt（日常小对话）保持标准预计算以确保首字精准度，只有长文件读取或多轮长对话（>4K）才触发 SpecPrefill 提速。 |
| **4.20 DFlash** | **`no`** | **坚决关闭**。解码阶段完全交给主模型内置的 Native MTP 负责，省去加载 DFlash 解码草稿模型的额外几 GB 内存。 |
| **4.21 Native MTP** | **`yes`** | **保持开启**。确保 1K~32K 全频段 13.5 ~ 19.6 TPS 的极佳解码流畅度。 |

* **优化后预期效果**：32K 下的 **TTFT 可从 303 秒缩短至 100~120 秒左右**，同时解码速度稳定维持在 **13.5 tok/s 以上**，峰值内存控制在 **25 GB** 以内，达成速度、内存与首字响应的最佳黄金平衡点。
