# Qwen3.6-35B-A3B-oQ4-mtp 最佳模型设置

> **生成平台**：OMLX 0.3.10  
> **运行硬件**：Mac mini 4 M4（内存 64GB 统一内存，SSD 2TB）  
> **使用场景**：Java 后端开发 / 前端开发 / 小程序开发 / 代码重构

| 属性 | 值 |

| **模型架构类型** | **混合专家模型 (MoE, Mixture of Experts)** |

## 2. 推理加速技术深度解析

### 2.1 Native MTP 原生多 Token 预测 — 推荐：✅ **开启**

| 项目 | 说明 |
| :--- | :--- |
| **为什么需要** | 本模型是原生包含 MTP 权重的版本。Native MTP 利用模型内部训练好的 MTP 预测头（MTP heads）并发预测多个未来的 Token，相比标准解码可实现约 1.5x 稳定提速。对于 MoE 架构，由于其单次计算量小（仅 3B），配合 Native MTP 可以达到极度流畅的生成体验。 |
| **核心优点** | ① **零内存额外开销**：不需要像 DFlash 那样额外加载几十 GB 的 MoE 对应草稿模型，极佳地保护了统一内存空间；② **稳定性极高**：内置预测头与主模型高度对齐，生成逻辑准确；③ **启动快速**：不需要额外加载第二个模型，单模型初始化即开即用。 |
| **核心缺点** | ① 提速幅度（约 1.5x）略低于理想状态下的 DFlash 投机解码（约 3x）。 |

### 2.2 DFlash 块扩散投机解码 — 推荐：❌ **关闭（可选开启）**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 外挂 35B MoE 专属的草稿模型（如 `z-lab/Qwen3.6-35B-A3B-DFlash`）进行投机验证，提供 3~4x 的解码加速。 |
| **为何关闭** | 35B MoE 模型原生解码速度已非常快（只激活 3B 参数），在开启 **Native MTP** 后，其 TPS 已经能满足日常极速编码需要。为了避免额外加载 `z-lab/Qwen3.6-35B-A3B-DFlash` 导致多占用数 GB 内存，推荐在日常开发中将 DFlash 关闭。如需追求极致的 150+ TPS 超高速，可以关闭 Native MTP 并开启 DFlash。 |

### 2.3 SpecPrefill 投机预填充 — 推荐：❌ **关闭**

| 项目 | 说明 |
| :--- | :--- |
| **原理** | 使用小模型计算超长 Prompt 预填充，降低首字延迟（TTFT）。 |
| **为何关闭** | MoE 架构由于单次计算量极小（仅 3B 参数），其原生预填充（Prefill）阶段本来就极快（32K 上下文下的 TTFT 仅为 69 秒左右，相比 27B 稠密模型缩短了 75% 以上）。因此，完全不需要开启 SpecPrefill，保持关闭能有效避免功能冲突并保持系统稳定。 |

## 7. 基准测试结果深度分析与性能瓶颈诊断

### 7.1 基准测试评估：为什么 Native MTP 比 DFlash 慢很多？

根据实测数据，在 `pp1024`（1K 上下文）下，`Qwen3.6-35B-A3B-oQ4-mtp` 开启 Native MTP 的生成速度为 **`69.2 tok/s`**，而同规格的 `Qwen3.6-35B-A3B-4bit` 外挂 DFlash 却能跑出 **`172.4 tok/s`** 的惊天高速度。

Native MTP 速度慢于 DFlash 的核心技术原因有以下三点：

1. **投机步长与预测头限制 (Speculative Step Size)**：
   * **Native MTP**：其加速原理是在主模型内部训练了固定的多预测头（MTP Heads），通常一次前向传播只能预测并验证固定数量的未来 Token（例如仅预测 2~3 个 Token），其加速比在理论上被锁定在固定的 **1.5x ~ 1.8x** 上限。
   * **DFlash**：这是一种更为激进的块投机解码。它外挂独立的草稿模型（Draft Model），在一次步骤中可以根据接受率动态预测大量的候选 Token（例如预测 5~8 个 Token），然后在主模型中一次性并发并行验证。
   * **MoE 模型的计算红利**：MoE 架构（Active 3B）的特性是单次验证矩阵计算极其快速廉价。DFlash 外挂方案预测的 Token 越长，越能充分压榨 MoE 主模型的高吞吐验证能力；而 Native MTP 预测的 Token 数量太少，导致 MoE 模型“吃不饱”，计算资源被闲置，无法发挥出高并发验证的硬件优势。

2. **MoE 架构专家切换与 MTP 头部计算开销**：
   * 在 MoE 模型中，激活 3B 参数需要经过路由器（Router）分配专家权重。在 Native MTP 的多头预测流程中，每个预测头也需要进行多次专家的激活与选择，增加了额外的调度算力开销。而 DFlash 的草稿模型通常是纯稠密的轻量模型，不需要进行专家路由，解码效率更高。

3. **内存占用与速度的终极权衡 (Trade-Off)**：
   * **Native MTP 方案**：解码速度虽然只有 `50~69 TPS`，但它在 32K 上下文下峰值内存仅占用 **`23.86 GB`**，且**零额外加载内存**，运行极其简单和稳健。
   * **DFlash 方案**：虽然能实现 `150+ TPS` 的爆发速度，但需要额外载入庞大的草稿模型，32K 上下文下峰值内存会迅速飙升并加剧系统分片风险。

---

### 7.2 针对 Qwen3.6-35B-A3B-oQ4-mtp 的优化建议

虽然 Native MTP 解码速度较慢，但在超长上下文（32K 及以上）的稳定性上，它表现得更为坚挺：
* **`pp32768` (32K)**：Native MTP 依然能维持在 **`35.1 tok/s`**，而 DFlash 在此长度下由于匹配率骤降及内存开销，已回退到 MoE 标准解码（`33.5 tok/s`）。
* **`pp65536` (64K)**：Native MTP 仍能维持在 **`23.9 tok/s`**。

#### 🌟 最佳部署配置建议：

* **方案 A：原生 MTP + SpecPrefill 极简稳定流（推荐用于日常与长上下文编码）**
  * **加速原理**：由于 **Native MTP 与 DFlash 完全互斥（不可同时开启）**，在启用 Native MTP 状态下，只能通过开启 SpecPrefill 进行首字延迟优化。
  * **具体设定**：保持 `Native MTP = yes`，`DFlash = no`。可选开启 `SpecPrefill = yes`（阈值 `8192`）。
  * **优势**：享受约 **50~69 TPS** 的平滑生成速度与约 **15~20s** 左右的超长文本首字即时响应（TTFT）。无需外挂解码草稿模型，32K 下峰值内存仅 **23.86 GB**，拥有无与伦比的内存安全感。

* **方案 B：DFlash 极致爆发流（可选切换）**
  * **具体设定**：必须将 **`Native MTP` 设为 `no`**（因二者互斥），同时将 **`DFlash` 设为 `yes`**，且将 `Draft Model` 指定为 **`z-lab/Qwen3.6-35B-A3B-DFlash`**，并配置 `Max Context = 20480`。
  * **优势**：在 16K 以内享受到 **150+ TPS** 的极致爆发生成速度。
  * **代价**：需多载入一个 DFlash 解码草稿模型，增加 4GB+ 内存占用。
