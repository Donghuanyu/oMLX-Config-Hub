# Qwen3.6-35B-A3B-4bit 最佳模型设置

## 1. 模型基本信息
* **模型名称**：Qwen3.6-35B-A3B-4bit
* **模型架构类型**：**混合专家模型 (MoE, Mixture of Experts)**
* **架构说明**：
  这是一款混合专家模型，总参数量约为 35B。MoE 架构的核心在于，每次前向传播生成 Token 时并不会激活全部 35B 参数，而是根据输入动态激活其中的约 3B 核心参数（A3B代表激活 3B 参数，Active 3B）。这使得它兼具“35B 级别大模型的表达与逻辑深度”和“3B 稠密模型的极快计算吞吐”。然而，在运行部署上，虽然它的计算开销小，但由于其全部专家权重仍常驻统一内存，它对内存的占用依然与 35B 模型相同（4bit 量化后约为 21GB），故依然依赖高统一内存容量。

---

## 2. 推理加速技术深度解析

为了在 Mac mini 4 (64G) 上实现最高的推理速度与表现，我们对各种加速技术做了深度适配：

### 2.1 DFlash 块扩散投机解码 (当前状态：启用)
* **为什么需要**：尽管 MoE 模型由于仅激活 3B 参数使得计算吞吐极快，但在本地单请求模式下，自回归解码速度的核心限制仍是内存带宽（Memory Bandwidth Bound）。DFlash 通过搭配一个 35B MoE 专属的草稿模型，并行提炼 Token 并在主模型中并发检验，大幅提升了生成效率。
* **优点**：
  * **突破瓶颈**：可为 35B 级别的大模型带来 3-4x 的端到端解码速度提速。
  * **无损验证**：草稿生成的文本经过 35B 主模型的严苛校验，确保逻辑推理和输出质量毫无退化。
* **缺点**：
  * **额外内存占用**：需同时在统一内存中载入 `z-lab/Qwen3.6-35B-A3B-DFlash` 这一草稿模型，需额外占用数GB的统一内存（在 64G 内存上完全能够轻松承受）。
  * **上下文长度受限**：当多轮对话上下文超过草稿模型限制（4096 Token）时，引擎会自动回退到标准自回归解码模式。
  * **单流处理**：适合单人做高负荷的开发工作。

### 2.2 SpecPrefill 投机预填充 (当前状态：禁用)
* **原理**：用一个极小的模型快速计算超长 Prompt 的预填充阶段（Prefill），然后将状态传递给主模型验证。
* **优缺点分析**：
  * **优点**：能够稍微降低长输入（如数万行的代码文件）的首字延迟（TTFT）。
  * **缺点**：与 DFlash 开启时相冲突，且增加额外的统一内存载入开销。由于本模型具有 MoE 的极快计算响应，因此在 64G 统一内存下关闭此项能换取更加稳定且少冲突的运行环境。

### 2.3 Native MTP 原生多Token预测 (当前状态：禁用，无法开启)
* **为什么无法开启**：当前您下载的 4bit 量化版本在转换过程中遗失了原模型声明的 `mtp.*` 权重张量。开启此项会导致 OMLX 报错无法启动。
* **优点**：不需要额外加载任何草稿模型，内存开销极低，可以直接通过主模型中内置的预测头实现约 1.5x 的稳定提速。
* **缺点**：模型文件转换要求高。
* **MTP 启用指南**：
  若想使用 Native MTP 提速，您需要：
  1. 下载官方完整未裁剪的带 MTP 头的 Qwen 官方 HF 原生仓库（例如带有 `MTP` 后缀的版本）。
  2. 使用最新版已合并 PR #990 和 PR #15 补丁的 `mlx-lm` 命令行重新进行 4bit 量化转换。命令如下：
     ```bash
     python -m mlx_lm.convert --hf-path <官方包含MTP权重的HF仓库路径> -q --q-bits 4 --mlx-path <输出路径>
     ```
     这能确保转换生成的 `.safetensors` 文件中包含 `mtp.layers.*` 相关张量，从而可被 OMLX 顺利识别。

## 3. OMLX 0.3.10 模型设置参数列表

## 4. 前端调用工具 (Roo Code / Cline) 专用系统提示词 (System Prompt)

请在前端工具的系统提示词配置中加入以下内容，从逻辑上杜绝假死和工具死循环：

```markdown
# 智能体运行准则（防止假死与工具死循环）

1. **工具调用截断保护**：
   - 当你读取文件或获取工具输出时，若发现内容不完整、缺少结尾括号或包含 [TRUNCATED] 字样，这表明工具返回结果受到了限制或文件过大。
   - **绝对禁止** 在不作修改的情况下连续重复调用同一个读取工具。
   - 解决方案：必须改用分段读取命令，或修改检索逻辑只读取特定行范围（如使用更具体的阅读或搜索工具），或者直接提醒用户由于文件过长需要由其精简或人工介入。

2. **自我复读阻断机制**：
   - 在任何情况下，如果你发现你的输出中包含与上一个回复高度重合的重复文本（自我复读现象），必须立刻停止生成，换用完全不同的措辞重新总结，或者向用户提出明确的问题以打断循环。

3. **代码设计与重构标准 (Java, 前端, 小程序)**：
   - **架构能力**：在进行 Java 开发时，严格遵循 Spring Boot 最新规范及阿里巴巴开发规约；前端开发需遵循组件化规范和单向数据流；小程序开发中必须注意内存占用与性能瓶颈。
   - **代码重构**：所有重构动作必须保证向下兼容。在重构核心模块前，先梳理调用链，并在代码中清晰注释核心逻辑。
   - **编码规范**：命名清晰（遵循驼峰或蛇形规范），异常处理禁止吞掉堆栈，所有工具类、接口类必须定义明确的职责边界。
```
